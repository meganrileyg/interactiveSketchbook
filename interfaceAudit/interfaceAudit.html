Interface Audit - Micro Project

Visibility of System Status

The page I am choosing to identify is my micro-project page. Once navigated to from the home navigation page, the user loses the index. The user must use their browser's arrows to go back, instead of easily navigating where they are in the sketchbook. An index could be added to the top of the page to fix this weakness. Additionally, when the user is interacting with the system, there is no real way of knowing how close they are to “answering” the phone, because although the circle grows and shows that you’re almost there, there is always the possibility of a dropped call to bring them back to the beginning. However, this is a purposeful design choice to add to the confusion of the system, so a status indicator would not necessarily benefit the purpose of the system.

Affordances and Signifiers

The actions available are the abilities to “answer” or “hang up” the phone by pressing either of the bracket keys. The actions are signified through text that shows up on the top of the screen, notifying the user what keys they should press in order to progress in the system. There is nothing else clickable on the screen, and all direct interactions from the user to the system are labeled. The dropped call feature is the only unclear feature, but it is not directly interactive, it simply happens to the user without their control.

Navigation and Mapping

To move through the page, the user clicks on the right bracket key. If they wish the exit (or “hang up”) they may press the left action key. The layout reflects the system structure, being that the sketch is the main asset. The title is the only thing above it in large letters, which introduces the system. You are introduced to the sketch, then the sketch itself, and below that are the instructions. Without navigating down to the instructions, the user will not know what to do, and could hesitate. Instructions on the sketch itself could help this, but it also could overwhelm the user right off the bat, and part of the system’s purpose is to cause hesitation and slight apprehension. Once the user navigates the instructions, the interaction is more or less predictable, with the exception of a potential dropped call which can happen randomly.

Constraints and Assumptions

The page assumes that the user can navigate, read, and follow written instructions. It assumes that the user understands the accumulation of the red circles and how they represent progress. The larger the circle, the closer to answering the call. The dropped call feature is mentioned in the instructions, but the user will not know when it may happen to them, and is therefore hidden.

Heuristic Violations

The first heuristic that my micro-project violates is the Flexibility and Efficiency of Use. The user only has two options for interaction: the left or right bracket to either hang up or answer the phone, respectively. Aside from this, the user has little flexibility in how to use the system. There are no shortcuts and no indications of whether a call may be dropped. No matter a user’s experience, the process will be largely the same. A fix for this could be a key shortcut that only experienced users will learn of, that answers the call immediately rather than filling the circle. This would eliminate the possibility of a dropped call.

The second heuristic is the Match Between the System and The Real World. In real life, if a user answers a phone, it is immediate. The user here may not expect to wait for the phone to be answered, and may get frustrated. Although already told that pressing the right bracket merely reaches for the phone (not answers it) users may expect immediate feedback. This could be fixed by a more clear indicator that the system is a game that the user must play, and that chance is involved.

Structural Revision Proposal

The instructions should be made more clear, either to the top or the sketch or above it, to enforce hierarchy and clearly navigate the user to a more clear interaction. The second would be to add an index to the top of the page so the user may clearly redirect themselves back to the interactive sketchbook homepage without becoming confused or lost. The third change would be to add a keyboard shortcut for experienced users to immediately answer the phone, should they choose to bypass the dropped call feature.
